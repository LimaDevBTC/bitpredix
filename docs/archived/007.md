### Problemas Identificados

#### 1. CONTENTION: Sponsor + Cron competem por nonces
O sponsor (para apostas de users) e o cron (para resolver rounds) usam o **mesmo endereço deployer**. Ambos broadcastam txs que consomem nonces da mesma conta. O sponsor tem KV tracker mas o cron não lê dele — cada um busca nonce de forma independente, criando conflitos e ghost txs.

#### 2. SELF-SPONSORED: Deployer apostando consome 2 nonces
Quando o deployer aposta, a tx é self-sponsored (origin == sponsor). Cada aposta consome 2 nonces, mas o KV tracker só registra +1. Isso cria gaps que travam a fila inteira.

#### 3. API DESSINCRONIZADA: Hiro extended API vs Node
O `/extended/v1/.../nonces` (possible_next_nonce) e `/v2/accounts/` podem retornar nonces que já estão ocupados por ghost txs no node real. Isso causa `BadNonce` e `ConflictingNonceInMempool`.

#### 4. BROADCAST FALSO-POSITIVO (já corrigido)
O `broadcastTransaction` retornava `{ error, reason, txid }` — o sponsor verificava apenas `'txid' in result` e reportava sucesso para txs rejeitadas.

### FASE 2: Separar nonces do Sponsor e do Cron (CÓDIGO)

**Problema**: Sponsor e Cron usam o mesmo endereço e competem por nonces.

**Solução**: O cron deve **ler e atualizar** o mesmo KV nonce tracker que o sponsor usa.

**Arquivo**: `app/api/cron/resolve/route.ts`

**Mudanças**:
1. Importar `getSponsorNonce`, `setSponsorNonce`, `acquireSponsorLock`, `releaseSponsorLock` de `@/lib/pool-store`
2. No início do cron, adquirir o sponsor lock
3. Em vez de buscar nonce via API, ler do KV tracker
4. Após cada broadcast, atualizar o KV tracker
5. No final, liberar o lock

```typescript
// Em vez de:
let nonce = await getNonce(address)

// Fazer:
const gotLock = await acquireSponsorLock(10000) // timeout maior para cron
try {
  const tracked = await getSponsorNonce()
  let nonce = tracked ? Number(tracked.nonce) : await getNonce(address)

  // ... processar rounds ...

  // Após cada broadcast bem-sucedido:
  await setSponsorNonce(BigInt(nextNonce))
} finally {
  if (gotLock) await releaseSponsorLock()
}
```

### FASE 3: Bloquear self-sponsored bets do deployer (CÓDIGO)

**Problema**: Deployer apostando cria txs self-sponsored que consomem 2 nonces e causam cascatas de problemas.

**Solução**: Não permitir que o deployer aposte — retornar erro no sponsor endpoint.

**Arquivo**: `app/api/sponsor/route.ts`

**Mudança** (após detectar `isSelfSponsored`):
```typescript
if (isSelfSponsored) {
  return NextResponse.json(
    { error: 'Deployer wallet cannot place bets (use a different wallet)' },
    { status: 403 }
  )
}
```

**Alternativa** (se quiser permitir deployer apostar): Criar uma segunda wallet derivada da mesma mnemonic (account[1]) e usá-la para apostas. Mas isso complica — melhor simplesmente bloquear.
